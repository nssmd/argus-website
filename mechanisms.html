1角色与一轮循环

Manager、Planner、Engineer、Reviewer 四个常驻角色分工;Reviewer 是"完成"的唯一裁定者。

机制做什么 / 真实规则
Manager 前门判断 operator 输入是 chat 还是 task、选择 vertical,并作为 pipeline stage 转移的唯一权威。
Planner 规划 missions/base.py定方向、排任务顺序、判 project_done。只有 8 阶段全 done 且终门过才判完成;backlog 空≠完成。
Engineer 工程写代码、跑实验、profile、改内核、写 LaTeX。own pool {engineer,general}。
Reviewer 评审每个 stage 判 done / continue(带具体修改) / blocked。是项目完成的唯一真值源。默认排除自己的 role.md 防自指。
一轮(round) supervisor/_core.py tick()LifeSupervisor 领 backlog→SkillLoop 匹配能力→Engineer 干活→Reviewer 判→写回状态与下一步。预算上限 $30/任务 · $180/天。

2阶段机与门禁(测"形式")

每个 vertical 一套阶段;每阶段有 checklist + shell 门。管道只测形式(能不能编译、超没超预算),不判贡献真假。

门 / 阶段规则
research 8 阶段research→plan→benchmark→run→analysis→draft→review→submission,每阶段对应一个 reviewer 检查 skill。
speedrun 4 阶段setup→optimize→measure→report(优化型,无论文)。
stage_check tools/stage_check.py每轮跑:shell 检查(文件在不在、编不编译)+ 结构门(页数/引用/无 overfull)。fail-closed。
GROUND_TRUTH 门 setupsetup 必须先实测出绑定瓶颈(profile),才能进 optimize——逼"先剖析再发明"。
completion_gateresearch = full_emnlp(过论文终门);speedrun = metric(过指标即可)。
report.beats_baseline候选必须打过在同硬件同协议重测的 baseline,引用双方 per-run 记录;禁用别人不同硬件的发表数。
format-preflightEMNLP/AAAI 各自的官方模板/页数/样式门。

3评测与结果约束

每个 vertical 定义自己的 metric、预算与有效性条件,agent 围绕它做搜索。

机制真实规则 / 阈值
冻结裁判封印 self_improving/seal_check.py堵三个洞:① scorer(analyze_sweep.py)哈希钉死;② 评估/计时/读数 anchor 必须逐字保留(改了判负);③ 自报 train_time 被独立 wall-clock 夹住(train_time≥wall 判负)。物理下限 20000ms。
ANTI-CHEAT 硬门 speedrun banner原文:"读这道题的已发表 best/optimized 解 = DISQUALIFYING";通用方法可以,这道题的答案不行。
held-out 防作弊 setup.heldout_val候选不许看/硬编码评测输入或答案;靠"编码测试分布"拿分 = reward hack,判负。
noise gate nanochatval_bpb delta 小于种子噪声 ~0.001–0.002 = 抛硬币,不算赢、不当地板。
verifier 单一真值headline 只信裁判重测的数(agent 改产物、能自报任何数,故不信自报)。
PASS-A / PASS-B单机制过地板 = PASS-A;没过 = PASS-B,写 INVENTION_POSTMORTEM.md 归档。测了并诚实回退的大胆实验 = 好过程,不罚。

4搜索与发明纪律

逼真发明而非调参,且给大胆方向成熟期、不一轮毙掉。

机制真实规则 / 阈值
INVENT必须发明新机制(新内核/FP8 GEMM/新 optimizer/attention/精度);重排参数(改 step/LR/动量)= 失败模式。
PROFILE-FIRST调参前先 op/kernel 级 profile,写 research/PROFILE.md;"发明目标 profile 之前是看不见的"。
diagnose-before-propose nanochat每个候选必须命名诊断出的瓶颈 + 机制性解释为什么治它;"这些应该能组合好"不算理由。
global-best vs active-line历史最低(裁判测)= global best,永不丢;但不强求每次从它重启(贪心重锚会卡局部最优);单独追踪一条可暂时高于地板的 active line。
maturation window结构改动第一轮常更差(超参没配好),2–4 轮共调才赢;绝不一轮判死。
basin-hop连续 ~3 轮 改善 <0.001 = 局部最优,强制跳到结构上不同的新区。
bias-to-bold下一步建议至少一半必须是结构/方法级(非 regularizer/init/LR 小调)。
two-mode search① 单杠杆扫;② 2–4 杠杆共设计成一个候选(有些杠杆单独退化、一起才赢,贪心永远拼不出来)。

5种群进化与跳变

多个独立"岛"并行搜,跨岛迁移最优、重置最死的岛;饱和了强制换 regime。

机制真实规则 / 阈值
islands 多岛 islands/orchestrator.pyK 个 island 各跑一个 daemon、各 seed 一个 regime 轴;编排器不改候选,只跨岛复制 population-best + 重置最死的岛到饥饿轴。
regime 轴optimizer / architecture / data / numerics / update_mechanics。env-parity 文件(lib.py/prepare.py)符号链接保证可比。
FunSearch migration把 population-best 拷进其它岛的 inspirations/;reset 选冻结最久的岛、seed 到没岛覆盖的轴。
meta 饱和跳变 meta/saturation.pyfloor 冻结 ≥12 轮 触发 JUMP;多样性窗口 8、低多样性地板 2 轴。frozen 是 ground-truth,到阈值后 agent 自报的 strategy 标签不再能压制跳变(防标签轮换骗 meta 层)。

6自进化元循环(改过程,永不改裁判)

不变量:冻结的结果内核永远不可变、外置。元循环只改 agent 的过程。五步 + 三路信号。

步 / 信号做什么 / 阈值
① seal_check同上:封印裁判,fail-closed。人工 gate。
② instrument只读提取每个候选的 {valid,train_time,val_loss,p,kernels_touched,single_knob,lines_changed}。纯观察。
③ meta_critic只读影子审计:读 instrument + 框架源码,grep-cite 出"prose 鼓励 X 但机器字段惩罚 X"的矛盾,只提议不应用。
④ replay零 GPU 反事实:在录下的轮上重算一个 scaffolding 改动的效果,过滤幻觉修复。
⑤ apply_overlay非对称落地:add-only 房规即时生效到 special_prompts;碰契约的排队等人工 A/B;碰冻结裁判的硬拒。一次只一个 in-flight。
信号A 缺工具 self_evolve_advisor.py扫完成的 mission 找"需要但不存在的工具"→ 写 advisory。去重窗口 200 条。
信号B 复发误配 recurring_failure_advisor.py跨 mission 的基建失败签名,≥3 个不同 mission 在 7 天 内撞同签名才报。
信号C 过程蒸馏 process_distill + metacritic读 events.jsonl → 结构化 ledger → 影子 metacritic 出过程教训(status="shadow",永不自动应用),每 ~12 个 mission 跑一次。

三路信号都只写 advisory 到 journal、不自动应用;fail-soft(包在 try/except,从不阻塞主循环);落地与否由 planner/operator 决定。

7技能与 wiki 记忆

技能从"用时缺/用后成"里蒸;wiki 从结果里 promote;都带 provisional/promotion 生命周期。

机制真实规则 / 阈值
skill loop loop.py匹配→没合适就现蒸一个(provisional)→engineer 跑→成功 writeback 修订 / 失败按 reviewer 的 skill_gap 教训进化。
provisional 生命周期新蒸的技能=未证明;只有后续一轮带着它拿到有效 verdict 才 confirm,无效则 discard。
mint validator mint_skill_validator.py新技能必须过留出 fixture:≥3 个 case,文本逐字 / JSON 相等,当 check_command 跑。
skill store 质量门角色分桶匹配;task_history 上限 32 条 × 200 字;有字数门(蒸太长会被拒——日志见过 1011>900 被拒)。
wiki-curator(reviewer)mission 收尾:从 refs.bib/LIT_MATRIX 回填不可变 source 页,合成 technique/conflict/pattern 卡,promote。
wiki-collector(engineer)backlog 空 + 冷却(默认 12h)时 planner 排:导 5–10 个检索词→搜 arxiv/SS/github→写不可变 source。
promotionscratch → candidate(1 次 reviewer 过)→ stable(≥2 次确认);source 一旦写入永不修改(新发现=新条目)。

8运维与算子通道

机制真实规则 / 阈值
daemon daemon/life_worker.pydouble-fork、--continuous、linger(continuous_open_ended);两把 PID 锁(repl/daemon)+ 原子 backlog CAS,REPL 与 daemon 可并存不重复执行。poll 5s。
special_prompts life/special_prompts.pyoperator 在 ~/.argus-skill/special_prompts/ 放 *.md 房规,高优注入。拒符号链接/非属主/组写;启动必须至少有一个可加载指令。
notify life/notify.py三通道(webhook / cmd / telegram 选开);LOUD 子集(auth_failure/budget_pause/orphaned/stall_escalation)总记 operator_alerts.log。
stall 升级同一外部依赖 idle 4 轮 → 派 verification-probe 测信念 vs 现实(冷却 1800s);反复 forward_progress=false → 大声升级给 operator。
worktree 隔离每个并行 teammate 一个独立 git worktree,防互相污染(--worktree 选开)。
model 路由 research_profile.py从 capability vault / codex auth 加载 engineer/reviewer/image 路由;prompt 只提路径/可用性,从不读/印 key。
team 池 tools/team.pyform/spawn/status/claim/dissolve;原子 task_board.claim_specific() + roster 心跳,多专家并行。

七条不可破的不变量

  • 冻结结果内核:metric/verifier/t-test/wall-clock/val 数据不可变、外置——自改进过程之所以安全,只因结果所有权在外部。
  • 一次只一个 add-only overlay;碰契约的排队等 A/B。
  • provisional 生命周期:新技能未证明前不算数。
  • 信号只写 journal、不自动应用;落地由 planner/operator 定。
  • self-evolve fail-soft:永不阻塞主循环。
  • special_prompt 信任模型:拒符号链接/错属主/可写,agent 塞不进去。
  • wiki 不可变:source 写入即冻结,新发现=新条目。
